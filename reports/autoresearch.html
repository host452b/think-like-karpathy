执行摘要

autoresearch 是一个 35 commits / 3 周 / 不到 3000 行代码的"仪式性 repo"——它本身是一份"让 Claude/Codex 过夜帮你跑 LLM 训练实验"的剧本。 真正的代码极少；真正新颖的东西是 program.md：一份写给 AI 代理看的自然语言"程序"。 人类不碰 train.py，agent 也不碰 program.md，这条分权线是整个项目的哲学支点。

从 git history 看三件事最清晰。 其一，首日就删除了 248 行的 spawn.sh（多-agent 协调脚本），项目从"多代理并行研究组"现实地收敛到"单代理 + 人在回路"。 其二，整个 3 月的 bug fix 全部围绕"代理失败模式"：NaN 不早退、无数据无限循环、崩溃不读 traceback、非 Hopper GPU 的 FA3 kernel——都是人类会轻松发现、但 agent 会呆坐在那里浪费一晚算力的问题。 其三，README/program.md 的改动占全部 commits 的近 70%，说明"上下文工程"本身就是此 repo 的第一等公民。

把它放回 2026 年开源 AI-agent 研究版图里，autoresearch 的定位很清晰：它不是 Sakana AI Scientist 那样端到端产论文的重型框架，也不是 AIDE / MLE-Bench 那样的学术基准，而是卡氏极简学派在"自主研究"这个命题上落下的又一颗子——最小化代码、最大化可读上下文、单 GPU 单文件、5 分钟时间预算、一个数字指标。它更像教学玩具和种子样板，而不是生产工具。

一、纵向分析 · 起源、诞生与演进

1.1 起源：karpathy 的极简学派在 2026 遇上 agent 时代

理解 autoresearch 必须先回到作者的前作谱系。从 2015 年的 char-rnn，到 2019 的 minGPT，到 2022 的 nanoGPT，再到 2023 的 llm.c 和 2025 秋季的 nanochat—— karpathy 做开源教材的方法论是一贯的：单文件、最少依赖、代码作为"可被一个人在一个晚上读完的文档"。2026 年初，这个谱系遇到了两件事：

  • agent 编码能力越过可用门槛。Claude Sonnet 4.5/4.6 与 Codex 已经能够在给定明确环境和指标的情况下，自主完成几小时到一晚上的 ML 工程迭代。
  • autonomous research 的热度。Sakana 的 AI Scientist 在 2024 就喊出了"AI 做研究"，2025 的跟进工作把这条线推得更远，但它们都复杂、昂贵、产物是"看起来像论文的 PDF"。

karpathy 给出的回答是反向的：不造"科学家"，造一个能过夜跑的小实验台；训练对象不是"通用 ML 问题"，而是他自己刚写完的 nanochat 的最小子集；目标函数是一个数字 val_bpb；时间预算是 5 分钟；所有复杂度都被挤压成两个东西——一个 train.py 和一个 program.md。

"One day, frontier AI research used to be done by meat computers in between eating, sleeping ... That era is long gone. ... This repo is the story of how it all began." — README 开头 Karpathy 自嘲式的虚构第 10,205 代。

这段开场白很关键：autoresearch 自我定位是"怀旧文物"，作者预设自己站在自主研究浪潮的起点，所以 repo 的价值不在"现在好用"，而在"埋下一根可以被放大十倍的轴"。这是理解后面所有设计取舍的钥匙——它允许很多事情"暂时不优雅"。

1.2 S1 · 首日的自我否定：spawn.sh 之死（2026-03-06）

initial commit b11d6f2 在 3 月 6 日一次性入库 10 个文件 / 2848 行。 里面有一份 248 行的 spawn.sh——autoresearch 原始设计是多代理并发：每个 agent 占一块 worktree、分支命名 autoresearch/<tag>， program.md 里第一部分就在教 agent "如果是 spawn.sh 启动的（多-agent），跳到第 3 步"。

但同一天内（3/6 晚间），spawn.sh 被删除。ae81d55、69eb7f9、以及稍早的 1e207aa "dam, erase experimental file from before that snuck through in my purge" 这三条 commits 像清扫现场： 

1e207aa  2026-03-06  dam, erase experimental file from before that snuck through in my purge
ae81d55  2026-03-06  remove spawn.sh reference from README (file was deleted)
69eb7f9  2026-03-06  cleanup more references to spawn.sh
                     program.md | 15 +++++----------

这是整个 repo 最重要的一次转向。多 agent 并发的"研究组"幻想在 24 小时内被主人亲手捻灭，取而代之的是单 agent + 人在回路的朴素形态。为什么？公开信息不足以百分百还原决策，但可以合理推测几个候选：

  • 早期 agent 并发的可调试性灾难：当两个 agent 在不同 worktree 各自改 train.py、共享 ~/.cache 的 data shards、又要避免覆盖对方 log，任何问题都要在多条分支上追，成本远超一个 GPU 并行的收益。
  • program.md 可读性的代价：带 spawn.sh 分支的 program.md 要区分"单/多"两种启动路径，自然语言程序最怕条件分支。删掉 spawn.sh 后，program.md 才真正做到了"一页剧本一个主语"。
  • "样板"优先于"工具"：只要 baseline 能跑，多 agent 可以让后人在 fork 里自行实现；核心 repo 要守住最小表面积。
信息缺口：3/6 单日内的 commit 间距只有几小时，没有外部博客或 tweet 记录作者当时的纠结。我只能凭 diff 推测——多代理并发写入结果表的同步问题是最大嫌疑，因为 program.md 里现在关于 results.tsv 的说明特别强调"不要 commit，留 untracked"。

1.3 S2 · 鲁棒性打磨：全是"agent 会呆坐的地方"（3/7–3/11）

这一周 11 个 commits 几乎全是在堵代理的失败模式，构成了"autoresearch 的真正难度不是算法，而是让 agent 不打断夜跑"这一主题的实证：

  • 03-07 01:31 PR#2 17b480a Marcin Bogdanski — add fallback FA3 kernel for non-Hopper GPUs。initial commit 默认用了 FlashAttention 3 / Hopper-only kernel，任何跑 A100/3090/4090 的 fork 都会 import 失败。这是"5 分钟预算"成立的前提：如果一半 GPU 都跑不起来，晚间实验台就没人能上。
  • 03-07 08:23 bd75534 merge "Fix agent crash blindspot by forcing it to read traceback"。这条 commit 名是整个 repo 最"工程侧"的一句——代理崩了默认只看最后一行错误，读不到真实 stack，program.md 被改成强制读 traceback。等于承认：代理的 debug 心智模型比本科生差，必须用流程兜底。
  • 03-07 多次 47ec1ad / bdbf0d0 / 6fdefa7 / 8a5c486 / ada84e5 连续小幅 tweaks program.md 与 README.md。最有意思的是 6fdefa7："instruct the agent to also read README, should be good context"——一条非常朴素的 prompt engineering 观察：给 agent 的 skill 文件里不给它链回 README，它就真的不会去读。
  • 03-09 c92bee5 "some docs on what to play with to make autoresearch better on smaller computers" + 068d93d "clarify that results.tsv should not be committed, leave untracked"——后者很关键：结果表留本地可以随时让代理改写/去重，一旦 commit 就产生分支冲突。
  • 03-09 b5ba8ac haosenwang1018 发现 NaN loss 没被 fast-fail 抓住；3/11 ebf3578 又有另一个 Contributor 独立修了同一处——两人独立踩到同一个坑说明这是实战高发雷区。最终 0be1e4f merge。
  • 03-10 09ebea4 Hugh Brown "Guard against infinite loop when no training shards exist"——如果用户没跑 prepare.py，代理会在空数据集上无限循环评估。这再次是"人类会立刻发现、代理不会"的典型症状。
观察：这 11 条 commits 里，有 6 条是非 Karpathy 的社区贡献者，而且都集中在"代理的鲁棒盲区"。这暗示 autoresearch 的早期核心用户就是在自己机器上夜跑的 hackers，他们踩坑回流速度极快。

1.4 S3 · 硬件生态扩散：一次"反 NVIDIA-only"的温和校正（3/7–3/16）

S2 的最大隐患（非 Hopper 全崩）被 PR#2 堵住后，Karpathy 随即做了几件事让 autoresearch 不被"仅 H100"标签绑死：

  • 7043095 add macos fork — 3/7。
  • 9264224 add notable fork mlx — 3/8，Apple Silicon 原生。
  • 2224cd7 "reshuffle readme a bit and link to tiny stories for apple silicon guidance" — 3/8，给低端设备用户铺了一条窄路（换数据集到 Tiny Stories）。
  • 513fe6f add AMD ROCm fork to notable forks section — 3/16，indianspeedster 的 PR#301。

对比一下 nanochat 在同期对 NVIDIA ClimbMix / H100 的深度耦合（见 nanochat 报告），autoresearch 刻意保留了"跨硬件"的可能性。原因推测：作为一个鼓励个人夜跑的 repo，硬件门槛每抬高一档，用户就流失一大块。

1.7 决策逻辑还原

哪些关键取舍有足够 diff 证据可以还原？

  • 5 分钟时间预算，而不是固定 token 数。README 解释："wall clock, excluding startup/compilation ... vocab-size-independent so architectural changes are fairly compared"。含义是：架构换算子/词表变化会让 token/sec 不可比，但墙钟时间不会——用"墙钟 + 指标"把"公平比较"拉回数据面。这是 karpathy 熟悉的 speedrun 语境的直接挪用。
  • agent 只能改 train.py。不只是"方便 review"，更重要是：prepare.py 既是数据也是评估，一旦允许动，agent 很快会"优化评估"而不是优化模型。这条红线是防 reward hacking 的物理护栏。
  • 不给 agent 装包的权限。ppt 上"只能用 pyproject.toml 里已有依赖"。等于把 agent 夹在"当前栈"里——不让它带 pip install 跑偏到另一种 ML 叙事。
  • VRAM 是"soft constraint"。不是硬阈值，而是"有意义增益可以适度吃显存"。这种软规则交给 agent 自己判，牺牲了可复现，换来了探索空间——对"夜跑"这种 open search 场景是对的。
  • "Simplicity criterion"写进 program.md。原文一句很值得抄下来——"A 0.001 val_bpb improvement that adds 20 lines of hacky code? Probably not worth it. A 0.001 val_bpb improvement from deleting code? Definitely keep." 这不是漂亮话，是agent 会把简单度偏置写进决策函数的硬指令——在没有这条的情况下，agent 会倾向无限堆 trick。

1.8 信息缺口与不确定点

无法在本 repo 内还原的问题：
  • autoresearch round 1/2 的实际 agent 对话 log 没有开源（也不在 nanochat 的 dev/LOG.md 里），所以"smear + backout"这两个架构改动的具体来源、代理提了多少个被否的实验——这些在公开 git 痕迹里看不到。
  • spawn.sh 删除前的"多 agent 并发"究竟跑过多少次、为什么被弃，作者没有 tweet 详述。
  • Apple Silicon / ROCm 的 fork 实际使用体量未知——notable forks 是被 karpathy 主动收录的三家，但没有 star 或 download 统计。

二、横向分析 · 放进 2026 年自主研究版图

2.1 赛道判定

autoresearch 所在赛道可以描述为："LLM 代理自主跑 ML 实验迭代"。这是 2024–2026 两年间逐步成型的一个小类，内部有明显分化。给出的 5 个参照：

项目类型选它的原因
Sakana AI Scientist端到端自主产论文最早把"AI 做研究"包装成完整叙事的项目
AIDE-ML (Weco)Kaggle-style 迭代代理同样是"agent 改代码跑实验"，但面向比赛基准
MLE-Bench (OpenAI)评测基座定义了"代理能不能做 ML 任务"的标准语料
OpenHands SWE-agent通用工程代理autoresearch 的 "program.md" 在哲学上是一种 skill，对标 SWE-agent 的 MCP/tool use

2.2 逐一对比

vs. Sakana AI Scientist

Sakana 的 AI Scientist（2024 年 8 月首发，2025 有 v2）目标宏大：从 idea → code → run → write → review 一条龙。 差异是能级与代价：AI Scientist 单次完整循环常需数十美元的 LLM 调用，产物是一篇自动写成的论文 PDF；autoresearch 单次是 5 分钟训练 + 一条 val_bpb 数字。

用户侧的真实体感：AI Scientist 在 Hacker News 和学术社区更多被讨论为"ML 版本的 demo"，论文里 reviewer 经常指出幻觉和"假阳性科学发现"。autoresearch 不试图写论文，只改 train.py——这让它几乎没有幻觉空间（跑不出来就是跑不出来，val_bpb 是一个小数）。

生态位判断：AI Scientist 像一台"一键产出论文"的黑盒整机；autoresearch 像一台自己组的小车床。前者野心大、覆盖广、可信度争议也大；后者刻意缩小战场、换来每一步都可复现。

vs. AIDE-ML

AIDE 同样走"代理改代码 + 实验循环 + 结果比较"的路线，但它面向 Kaggle/MLBench 式的表格或视觉任务，核心价值是在有 leaderboard 的任务上推 agent 性能。

autoresearch 的场景正好反过来——不是 benchmark shopping，而是一个单一的、由作者控制的训练任务。 这种窄聚焦有两层好处：（1）reward hacking 面积小，agent 不容易钻评测漏洞；（2）产物可以直接回注 nanochat 主线。 代价是：autoresearch 无法像 AIDE 那样做通用 ML 工作流代理。

用户口碑上，AIDE 在 Kaggle grandmaster 圈子有真实使用（尤其表格赛）；autoresearch 目前更像 karpathy 粉丝圈的研究玩具，未见任何"职业选手"在里面打比赛。这是定位差异而非优劣差异。

vs. MLE-Bench

MLE-Bench 是 OpenAI 2024 年放出的 75 道 ML 任务评测集，定位是"衡量代理的 ML 工程能力"。它是基座而不是代理——你拿你的代理去跑它。

autoresearch 与它形成奇妙的互补而非竞争：MLE-Bench 提供广度评测，autoresearch 提供深度单点；前者问"agent 到底能不能做 ML？"，后者问"agent 能不能把一个具体项目往前推一寸？"。 若未来有人想拿 autoresearch 的 delta 作为单点评测加入 MLE-Bench 类基准是完全合理的。

vs. OpenHands SWE-agent

OpenHands 是通用工程代理框架，里面最成熟的是 SWE-agent（SWE-Bench 上的重量级选手）。 哲学分歧极深：OpenHands 通过 MCP tool protocol / 函数调用 / 环境沙箱构建能力，autoresearch 反过来——不上任何 tool framework，就一个 program.md。

更本质的对比是：OpenHands 把代理能力堆在"tools × LLM"的乘法上；autoresearch 把代理能力堆在"skill-as-markdown × 人类干预"的乘法上。前者工业化、能扩展；后者极简、可读。 autoresearch 的隐含赌注是：2026 年的 frontier LLM 已经足够聪明，只要给它一份好 prompt 和一个窄场景就行。 这个赌注目前在 nanochat leaderboard 上赢了两次（round 1、round 2），但还没有被第三方广泛复现。

vs. modded-nanogpt

modded-nanogpt 是 Keller Jordan 等人 2024–2026 持续维护的"人类 speedrun"赛道，每个月都有新的 record。autoresearch 是它的 AI 代理对偶版——两者都共享"固定时间 + 单指标"的竞速哲学。

karpathy 在 README 里明说受 modded-nanogpt 启发。差异在于：

  • modded-nanogpt 强调 leaderboard 是公开 PR，人脑优化；
  • autoresearch 强调 leaderboard 是单机夜跑，agent 优化；
  • 两者的评测维度类似但不完全可比——modded-nanogpt 是 val loss@fixed iters / 时间；autoresearch 是 val_bpb@5min wallclock。

这对组合的完整叙事其实是：karpathy 在同时押两条线——"人的极限"和"agent 的极限"——而他显然认为下一个 breakthrough 更可能来自后者。

2.3 用户视角（可追溯来源）

autoresearch 的 GitHub 讨论区和 PR（#2, #301, #342 等编号告诉我们 issue/PR 总量已到 300+）显示的三条高频反馈：

  • "非 H100 如何跑"。FA3 fallback PR（17b480a）和 ROCm / MLX notable forks 就是这条反馈的具象化。用户真实的使用路径是"先尝试把 baseline 跑起来"，这比"让 agent 刷 bpb"低得多的门槛，反倒成了第一大故事线。
  • "agent 卡住了怎么办"。traceback 强制读取、NaN 早退、数据缺失检测，三条 commits 全部指向这个痛点。结论：autoresearch 的实际用户画像不是研究员，是晚上想让 agent 帮自己省两小时的工程师。
  • "我能改 program.md 吗？"。Karpathy 在 README 里明说"program.md is essentially a super lightweight 'skill'"——这回应了大量"我想让我的 agent 也这样"的请求，顺手把 skill-as-markdown 的概念推了一把。

autoresearch 占的是"最窄最深 + agent 主导"那个角落。它不与 OpenHands 争工程能力的广度、不与 Sakana 争论文的覆盖、不与 AIDE 争评测赛场，而是把"单一 repo 的极小改进"这件事做到了代理就能稳定跑一晚上不崩的程度。

2.5 机会与风险

  • 最大机会：作为"skill-as-markdown"的样板 repo 被大量复用。任何人想让 agent 帮自己迭代一个私有项目，autoresearch 的结构（两个 md + 一个可变 .py + 一个只读 prepare.py + 一张 results 表）可以直接抄。
  • 最大风险：窄场景的天花板。nanochat 的 speedrun 已经从 3.04h 压到 1.65h（autoresearch round 2），但每一 round 的边际收益递减——当 baseline 被榨干后，agent 要么要被放到更大的任务空间里（此时就不是 5 分钟预算的窄场景了），要么价值就止步。
  • 次要风险：frontier LLM 版本的强依赖。整个范式依赖于 Claude/Codex 在 2026 年的水平；一旦模型换代，program.md 可能要重写才能继续工作。

三、横纵交汇总结

3.1 路径依赖判断

autoresearch 的 35 条 commits 给它建立了三个结构性特征，未来很难反转：

  • 极简偏执。spawn.sh 首日被删、后续所有扩张诱惑（多 agent、workflow 引擎、observability）都被挡住了。这意味着它永远不会演化成 OpenHands 那种全栈工程代理，但也意味着它会长期保持"可被一个人读完"的特性。
  • 人改 md / agent 改 py 的分权宪法。这条线是 program.md 和 README 的高频修改建立起来的"程序文化"。换 agent、换 LLM 不会打破这条线；但如果哪天 agent 被允许改 program.md 自身（self-improvement），整个合同就失效——到那一刻 autoresearch 已经不是现在这个项目了。
  • 硬绑定 nanochat。train.py 本身是 nanochat pretraining 的单 GPU 蒸馏版，autoresearch round 1/2 的改进被直接回吞进 nanochat。这让 autoresearch 不能独立漂移——nanochat 换了 model 或 dataset，autoresearch 必须跟进；反过来 autoresearch 也成了 nanochat 的预研仓。

3.2 差距的可追赶性

把横向对比里暴露的差距拆成两类：

差距可追赶性推理
覆盖广度（vs OpenHands / AI Scientist）不可追赶这是刻意放弃的维度，不是能力不足。追上去就不是 autoresearch 了。
基准化（vs MLE-Bench / AIDE）部分可追赶autoresearch 可以被包装成"单任务评测"加入更大基准，但 repo 本身不会做。
社区 leaderboard 热度（vs modded-nanogpt）可追赶当前只有 karpathy 一人把 autoresearch 跑进了 nanochat leaderboard。一旦第二、第三个人加入，就会形成竞速。
硬件覆盖已在追赶FA3 fallback / MLX / ROCm 三条线已经铺开，接下来是社区 fork 的维护度。
代理鲁棒性可追赶NaN 早退、traceback 强制读取这类模式，积累一段时间后会沉淀成可复用的 "agent runtime policies"。

3.3 决策建议

（假设我是负责 autoresearch 路线的人。）

  • 最应该做的一件：把 program.md 抽成可复用的 "skill-as-markdown" 样板文件集合，并给出"把你自己的项目接入 agent 夜跑"的 5 步模板。当前 autoresearch 的最大杠杆不在 train.py 再压 0.01 val_bpb，而在让第二个、第三个项目用 autoresearch 的"宪法"运行起来——这是哲学的放大效应。具体路径：把 program.md、prepare.py 的 read-only 语义、results.tsv 的 schema、traceback 强制策略，合并成一个独立的 autoresearch-kit，autoresearch repo 自己只保留"应用样例"。
  • 最不应该做的一件：不要给 agent 开 program.md 的写权限。一旦 agent 能改自己的 skill 文件，"人改 md / agent 改 py" 的分权宪法就垮了，整个 repo 的可审计性瞬间归零。Self-improving agents 在 2026 是热门议题，但把它引入 autoresearch 等于用哲学资产换一个别处已经有很多人在做的短期 demo。退一步，即使探索也只在 fork 里做，主干必须守住。
