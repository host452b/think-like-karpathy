执行摘要

hn-time-capsule 是 karpathy 一次 "双次坐下、中间睡一觉" 的 vibe coding——第一次坐下（12/9 13:27–19:38，约 6 小时）写完 v0 到 IMDb 排名，10 条 commits；第二次坐下（12/10 07:42–08:59，约 1.5 小时）做清理、加 hero 图、发博客。一共 36 小时、2000 多行 Python、99% 由 Claude Opus 4.5 生成。

项目本身做一件小但迷人的事：抓 10 年前同一天的 HN 首页，把每篇文章和评论丢给 LLM，让 LLM 凭今天的后视镜给当年的评论者打分（A+ 到 F）。最后跨日累积，产出一张 "hall of fame" ——哪些账号长期预见准确、哪些长期打脸。

从 git 提交史里看三件最值得学的事。 其一，架构是典型 5 阶段磁盘缓存 pipeline：fetch → prompt → analyze → parse → render，每一阶段的输入输出都是文件，任何阶段可以单独重跑/清空——这让一个"要花钱跑 LLM"的项目在调试期几乎免费。 其二，最有信息量的 commit 不是任何 bug fix，而是同一个 prompt 被"tune"了两次（76da7d5 "tune prompt again"）——每次都不是改逻辑而是改输出结构，为了让 parser 更好读。这暴露了 LLM 应用工程的本质：程序的"接口"长在自然语言里。 其三，CLAUDE.md 被作者主动删掉（bdde346）——"people are free to init their own i think"——作者把"你自己的 agent 指令"视作私有配置，而不是项目资产，这和 autoresearch 里 program.md 作为公共资产的立场形成有趣对照。

这类 repo 在 2026 年已属"一日工具"：没有竞品、不打算维护、但范式本身可复用。它最值得学的不是代码，是"36 小时内把一个小而野的点子做成能发 blog 的完整品"这个节奏感。

一、纵向分析 · 36 小时内的造物史

1.1 起源：一个一直被想但一直没被做的念头

让 LLM 穿越回去审视互联网的旧内容，这个点子在 2023 GPT-4 发布后就在社交媒体反复出现——"让 ChatGPT 读 2010 年的 reddit 预测看谁最准" 之类的短推文时有时无。问题是：手工复制几篇帖子喂给 ChatGPT 门槛很低、但把"所有人、所有天"系统跑一遍的 pipeline 门槛不低，而写这种 pipeline 的人又大多不是对 HN 社区情怀深的人。

karpathy 正好是 HN 早期用户、有足够的 OpenAI API credit、也有一个 Opus 4.5 订阅可以"代写"。他在 blog 里说得很直接："The more general and interesting idea is that of LLMs automatically scouring human artifacts from the past and synthesizing them into knowledge and insights."（这才是更大的命题：让 LLM 去扒人类留下的旧产物，沉淀出洞见）——HN Time Capsule 是这个命题的一个可见的锚点。

"99% of this repo was vibe coded in a few hours with Opus 4.5. Code is provided as is and I don't intend to support it." — README "Vibe code alert" 段落

1.4 最值得细看的一个 diff：76da7d5 "tune prompt again"

这次 prompt 改动不改"让 LLM 做什么"，只改"让 LLM 怎么输出"。关键 diff： 

-Let's use our benefit of hindsight now:
+Let's use our benefit of hindsight now in 6 sections:

-1. What ended up happening to this topic?
-2. Give out awards for "Most prescient" and "Most wrong"
-3. Mention any other fun or notable aspects
-4. Give out grades to specific people
-5. At the end, give a final score
+1. Give a brief summary of the article and the discussion thread.
+2. What ended up happening to this topic?
+3. Give out awards for "Most prescient" and "Most wrong"
+4. Mention any other fun or notable aspects
+5. Give out grades to specific people
+6. At the end, give a final score ...

-pattern = r'(?:^|\n)(?:#+ *)?Final grades\s*\n'
+pattern = r'(?:^|\n)(?:\d+[\.\)]\s*)?(?:#+ *)?Final grades\s*\n'

三件事在同一个 commit 里发生：

  1. 显式告诉 LLM "6 sections"——因为观察到 LLM 会偷懒合并或跳过某一节；把总数告诉它，它更服管教。
  2. 加了一节"先总结文章+讨论"——也就是说之前跑下来 LLM 的预测点评有时脱离原文语境。加一节 summary 是把 LLM 的注意力先拉回原文。
  3. parser regex 加了"可选的数字+点或括号"前缀——因为 LLM 现在会输出 5. Final grades 而不是单独的 Final grades，如果 parser 不认就会漏一整批数据。
这一条 commit 比任何 README 都更好地示范了 "LLM 应用工程" 的三件套：①控制输出结构（让输出更可 parse）②加入 grounding（先 summary 再评价）③parser 对格式漂移保持宽容。做过 LLM-serving 的工程师都懂这三点，但把它们浓缩在一条 9 行 diff 里非常罕见。

1.5 核心架构：5 阶段磁盘缓存 pipeline

pipeline.py 被高频修改（11/15 commits 都动了它），但核心结构从 v0 起就稳定：

阶段命令输入输出是否花钱
1. fetchpython pipeline.py fetch日期文章 + HN 评论 JSON + 文章正文 HTML否（网络）
2. promptpromptstage 1 输出每文章一份 prompt.txt否
3. analyzeanalyzeprompt.txt每文章一份 response.txt（LLM 输出）是（API $$）
4. parseparseresponse.txtgrades.json + score.json否
5. renderrenderstage 4 输出HTML summary + hall of fame否

这个设计的精妙在于：每个阶段的产物是"可失效的缓存"。改了 prompt（stage 2）→ 重跑 3-5；改了 parser 正则（stage 4）→ 只重跑 4-5，第 3 阶段昂贵的 LLM 调用完全不动。这是 LLM-heavy pipeline 的标准打法，但能在 v0 initial commit 里就把 stage boundary 切对，是工程功力。

stage_clean（22dec7a 加的）配合这个结构：允许选择性抹掉某阶段、甚至某一篇文章的某阶段 cache。这个 subcommand 比任何 README 都更清楚地讲出了作者的调试心智——"我已经预期到我会反复调 prompt、调 parser，所以我先把'局部失效'这个能力加进来"。

1.6 决策逻辑还原

  • 为什么用磁盘缓存而不是数据库？磁盘 = grep / cat / 肉眼检查；数据库 = schema migration + 客户端。对 36 小时一次性的工具，磁盘完胜。
  • 为什么 Markdown 而不是 HTML 给 LLM？ clean_html_to_text 把 HN 评论 HTML 扁平化成 Markdown——LLM 在 Markdown 上的 tokens 更紧凑，原文里的 <p>/<div> 对评价预见性毫无价值。
  • 为什么要有 IMDb-style interestingness score？ 如果不评分，render 阶段要么按 HN 原始排名（基于当年的 upvote），要么随机——前者让 10 年前的大热门永远占上，后者太乱。让 LLM 打一个 0-10 的"后视镜趣味度"，把评价能力的一部分也交给 LLM。
  • 为什么 hall of fame 要压缩？ fe5b01a 说"shorten the disk size a bit"——跨多日的全量输出会爆盘、也太大传不动。压缩的同时把 HTML 里不必要的原文字段剥掉，只留 grade + 引用片段。
  • 为什么删 CLAUDE.md？ 这是一个微妙的社交决策。作者的态度是 "你让你自己的 Claude 看你自己的 CLAUDE.md 就行，不要跟着我的私有配置走"。这把 agent-instruction 划入了"用户 dotfile"的范畴，而非项目资产。

1.7 信息缺口

  • initial commit 之前到底有多少轮 Claude 对话产生这 2073 行，没有 git 证据。作者 README 说"a few hours"，但那几小时的生成过程只能靠推测。
  • "tune prompt" 实际的 A/B 对比没有开源。不知道旧 prompt 漏了多少个 Final grades 块。
  • 该工具的 API cost 没有披露，但从 `max_workers=5`、`model="gpt-5.1"`、每日约 30 文章 × 每文章万字级 prompt 推算，一日数据≈几美元量级。

二、横向分析 · 为什么这个赛道里几乎没有"竞品"

2.1 类型判定：场景 A（无直接竞品）

"10 年前 HN 首页 + LLM 后视镜评分 + 跨日账号聚合"这个组合，在公开互联网上找不到第二个。这并不奇怪——以下 4 个相邻位置一起解释了为什么这个生态位长期"空着"：

相邻方案定位与 hn-time-capsule 的差距
Wayback Machine + 人工 Ask HN纯档案 + 人类怀旧没有结构化评分；每十年有人发 "what did we get right 10 years ago" 但都是手写感想
The Pudding / 538 数据新闻式回顾编辑驱动的精选长文单篇成本高（人 + 可视化设计），不可能按天批量产出
ChatGPT / Claude 手动 prompt随想随问一次只能看一篇；无聚合；无持久化
niche HN scrapers（hnhiring、hnbooks）结构化抽取 HN都是"现在有什么"（工作/书单），不是"过去预测得对不对"

2.2 为什么这个生态位空着？

四个原因叠加：

  1. LLM 能力门槛直到 2024 下半年才跨过。给 LLM 一篇长文 + 100 条嵌套评论 + 要求按结构输出成绩单 + 给出 0-10 分，需要（a）大 context window（>32k），（b）结构化输出可靠性（JSON-mode / constrained decoding），（c）对普通社区语气的 robust 理解。这三项合格的模型，真正普及是 2024 Q4 以后。
  2. 成本曲线刚好。用 gpt-5.1 跑一整天的 HN 首页只要一顿饭钱；2023 年用 GPT-4 跑就要 $100+。成本必须先降到"可以每周为你自己刷一遍玩"，这种玩具才会被做。
  3. HN 这个社区不太容易被"研究"。HN 不提供全量 API 的历史快照，frontpage 渲染也在变（URL 如 front?day=2015-12-09 是 HN 很晚才加的特性）。只有对 HN 足够熟的人才知道这个入口，并且有动机写一个针对性 scraper。
  4. 更深层的抑制因素：结果尴尬。HN 老用户发现自己的旧帖被 LLM 打 F，从社区治理角度不算舒服。这是 karpathy 的 "Vibe code alert + 不维护" 声明的隐含保护——以"玩笑一日 repo"的姿态发布，社区接受度最高。

2.3 场景 A 下的竞争者来自哪？

几乎可以肯定，未来 12 个月内出现以下变种中的一种或多种：

  • 平台内嵌。如果 HN 自己（或任一社交平台）推出"retrospective score"视图，这类 repo 瞬间失去独立价值。可能性中等——HN 向来保守，反而 Reddit/Twitter 更可能先做。
  • 多社区复制。把同一套 pipeline 套到 Reddit r/technology 2015、Twitter tech Twitter 2015——pipeline 的 5 阶段可无痛复用，fetch 阶段换 adapter 即可。
  • "预测市场 + 后视镜"。有人会做 Metaculus / Polymarket 的历史对账——这比 HN 评论远远更严肃，是真的社会价值。
  • 学术化：一篇"LLM-as-retrospective-judge"的方法论论文，研究 LLM 作为历史判官的 bias/日期偏袒。

2.4 用户视角

repo 本身只有 karpathy 一个贡献者、无社区讨论，但 bearblog 的博文 下有零散评论，HN 上同步发布的讨论帖也出现了两类真实反馈（基于可追溯的公开评论）：

  • "能不能按用户名聚合多年？" — 这正是 hall of fame 已经做到的；提问者显然没读完 README，但这暴露了用户本能地想从中找出"常青预言家"——这个 instinct 比项目本身重要。
  • "为什么不做 Reddit？" — 非常高频的追问。答：HN 是 karpathy 的主场社区，Reddit 需要另做 fetch adapter，重点是一天内做完。
  • "LLM 会不会有 hindsight bias？" — 有人指出，LLM 本身就读过很多 2015 以后的新闻，它给 2015 年预测打分天然带上帝视角。这是 repo 的方法论盲区——README 里没承认这一点。

三、横纵交汇总结

3.1 路径依赖判断

这个 repo 刻意避免了路径依赖——作者公开宣告 "I don't intend to support it"。但从结构上，它留下了三处"可复用模板"：

  • 5 阶段磁盘缓存 pipeline。可以直接抄到任何"抓取 → 处理 → LLM → parse → 呈现"的项目里。
  • prompt 设计的输出契约：显式编号 sections + 用硬性前缀（"Final grades" / "interestingness score:"）锚定机器可读部分 + parser 对格式漂移宽容。这个三件套在很多 LLM 应用里都值得直接套用。
  • "vibe + 声明不维护"的发布姿态。一个 36 小时的 repo 如果不加这个声明，维护者会被议题淹没；加了以后，社区预期对齐，批评也变温和。

3.2 差距可追赶性

对比位差距性质可追赶性
vs. 平台内嵌获客 + 影响力不可追赶（一旦 HN 或 Reddit 官方做，这类 side project 不可能对抗）
vs. 严肃 retrospective（Metaculus 对账）方法论严谨度部分可追赶——如果加入 baseline LLM 的 hindsight-bias 评估就可以升格
vs. 数据新闻（Pudding 式）叙事深度 / 视觉品质不可追赶，且不必追——本 repo 是生产工具，不是出版物
vs. 多社区 fork覆盖广度可追赶，但作者明确不会做，只能靠社区 fork

3.3 决策建议（如果我是要继续做这个项目的人）

  • 最应该做的一件：把 hindsight bias 校正加进 prompt。加一步让 LLM 先列出"如果它不知道 2015 后的历史它会同意哪些评论、反对哪些"，然后再在这基础上打分。这能把 repo 从 "好玩" 抬到 "有方法论立场"——同时也把批评（"LLM 作弊"）正面吸收进产品。成本：prompt 多一节 + 1 次 LLM 调用。价值：首次可以对外引用的 retrospective 评分方法学。
  • 最不应该做的一件：不要做发现性的功能扩展（比如"按话题分组找长期预言家"、"自动找出最被低估的冷门帖"）。这些想法都很香，但每一个都会再花 36 小时，而且引入社区治理风险（被 HN 老用户围攻）。与其扩张，不如把 5 阶段 pipeline 抽成一个可 pip install 的小库，让别人在 Reddit/Twitter/Metaculus 上复用它——让 repo 成为 pattern 而非产品。
