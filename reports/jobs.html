jobs 是一个刻意"去中心"的数据可视化 repo——BLS 官方 342 个职业、每格面积等于就业规模、颜色可在"BLS 增长预期 / 中位薪酬 / 教育门槛 / 数字 AI 暴露度"四层之间切换。 它被作者亲自从"AI 对就业影响的一份分析"重命名、降级、包装为"一个探索 BLS 数据的开发工具"——这次定性反转就藏在 1cca284 "update the readme to minimize misunderstanding of this repository" 这一条 commit 的 diff 里，是整个 repo 最有研究价值的事件。

从 git 提交看三件事。 其一：发布节奏是 初次发布 6 小时 + 次日再坐 4 小时——initial commit 在 3/14 13:42 一次性入库 656,087 行（其中绝大部分是 html/ 原始 BLS 页面 40MB + 342 行 scores.json + 2054 行 occupations.json），之后 10 条 commits 全部是小迭代。 其二：3/15 有一次非常明显的"项目定位逆转"——从 AI-exposure 叙事主导 → AI-exposure 只是 4 个可视层之一。这是典型的被互联网舆论/反馈 push 回工具定位的案例。 其三：prompt.md 的独立出现（7fa0be6 "Add prompt.md: all data in one file for LLM analysis"）是一个非常新兴的模式：repo 显式把 LLM 当作一等受众——"你不用跑我的前端，把这个 45k token 的 prompt 贴进 ChatGPT 就能和数据对话"。

一、纵向分析 · 从"结论陈述"到"探索工具"的 31 小时

1.1 起源：2026 年，AI 就业讨论已经嘈杂到什么程度？

把 2026 年 3 月的舆论环境具体化一下就能懂 jobs 为什么这样发布。此时：

  • Frey & Osborne 2013 的"47% of US jobs at risk of automation"已经被讨论了 13 年，原作者自己都在新论文里不再用那个数字；
  • OpenAI 2023 年 "GPTs are GPTs" 研究估计 80% 美国工作者有至少 10% 任务暴露；
  • Anthropic Economic Index（2024 起每季度发布）基于 Claude 的真实对话分布，给出"按职业的 LLM 使用频次"——这已经是业内公认最贴近真相的数据源；
  • McKinsey / Goldman Sachs / WEF 每几个月就出一份 AI-对-就业的白皮书，数字从 300M 到 7.2M "可能失业" 不等；
  • 社交媒体一面倒："AI 会淘汰程序员" / "AI 让程序员更强"两派每天对喷。

在这个环境里发布"一个 LLM 打分的 AI 暴露表"，任何读者本能地会把它当作第四份"预测报告"。这就是 jobs 面对的信号噪声比——而它用"降级为工具"的方式走出了一条与众不同的路。

1.5 另一个值得单独讲的 commit：7fa0be6 prompt.md

prompt.md 把下列内容塞进一个单一 Markdown 文件（约 45k token）：

  • 聚合统计（平均 AI 暴露、各 tier 职业数、按薪酬/教育的分布）
  • BLS 增长预期汇总
  • 342 个职业的名称、SOC 代码、就业规模、薪酬、教育、AI 暴露分 + 评分 rationale
  • 每份 rationale 有 2-3 句 LLM 写的解释

README 写得很明白："This lets you have a data-grounded conversation about AI's impact on the job market without needing to run any code."

这是一个值得被单独命名的新兴设计模式——不妨叫它 "single-file conversational dataset"：

  • 传统数据集给机器：JSON/CSV/Parquet，预设有代码处理。
  • prompt.md 给 LLM：Markdown + 人话注释 + 已经聚合好的总结，预设有人会把它复制粘贴。

在 LLM 文化里，这类 repo 越来越多（例如 nanochat 的 dev/LOG.md、autoresearch 的 program.md、LangChain/llama-index 的 "chat-with-your-data" 样板）。但 jobs 是我见过把它单独列成一个面向 LLM 受众的 artifact 最干净的样例：前端给人看，prompt.md 给 LLM 看，两条路都通。

1.6 核心架构 · 6 阶段磁盘缓存 pipeline

#阶段脚本产物花钱

5build site databuild_site_data.pysite/data.json否

和 hn-time-capsule 的 5 阶段 pipeline 是同构的。更值得单独讲的是 score.py 里的 LLM 评分 rubric： 

0-1: Minimal. Physical/outdoor/unpredictable. Roofer, diver.
2-3: Low.     Mostly physical/interpersonal. Electrician, nurse's aide.
4-5: Moderate. Mix. RN, police, veterinarian.
6-7: High.     Predominantly knowledge work. Teacher, manager, accountant.
8-9: Very high. Almost all-digital. Software dev, designer, translator.
10:   Maximum. Routine digital. Data entry, telemarketer.

这把 "AI 暴露"的语义锚到"工作产物是否原生数字"——不是自动化概率、不是 job displacement、不是 wage impact，而是"是否能完全坐在电脑前完成"。 这个定义很窄，但窄得可操作，让 LLM 打分有一致性；同时它避开了 "被 AI 替代概率" 这种必然踩雷的预测。定义的克制是这个项目技术上最精妙的一个点，比任何前端或 pipeline 都重要。

1.7 决策逻辑还原

  • 为什么用 Playwright 非无头而不是 requests / aiohttp？ 因为 BLS 在 OOH 页面对 bot 检测比较严。非无头 Playwright 开销大但成功率高——对 342 页一次性抓取是合适 trade-off。
  • 为什么用 Gemini Flash 而不是 Claude / GPT-5？ 成本。342 × Markdown 每页 ~3k token，配 rubric prompt ~1k token，Gemini Flash 单次约 $0.0002；用 Claude Opus 就是 10-50 倍。这是"玩具成本"和"可重跑"之间的直接交易——选便宜模型让这个工具可被任何人分叉重跑。
  • 为什么只打 0-10 一个维度？ 多维度会让前端配色策略变复杂、rubric 也难对齐。一个数字 + 一段 rationale 是"可视化友好 + 不完全丢失上下文"的最小表达。
  • 为什么 "outlook vs exposure" 视图在 3/14 就加了？ 因为作者自己已经意识到"单独看暴露会误导"——在第一天就埋下了"需求增长会抵消暴露"的对比视角，到 3/15 的 README 改版才把这个自觉显式化。
  • 为什么把 scores.json 和 occupations.json 都入库？ 任何人 fork 后无需重跑 score.py 即可得到同样的视图；同时 git diff 能直接显示"换 prompt 后分数怎么变"。这把"可重跑性"和"可 diff 的结论"打了个组合拳。

1.8 信息缺口

  • initial commit 前的开发时间未知。README 没说"vibe coded"但按结构看必然用了 agent 辅助。
  • "被定位反转"的外部触发点（某条 tweet / HN 评论）没有被记录。只能从 commit 节奏和 diff 方向推测。
  • Gemini Flash 的实际 scoring 偏见没有被独立评估。换模型跑一次、对比 variance 可以回答"这分数有多稳"，但作者没放。

2.2 逐一对比

vs. Frey-Osborne (2013)

Frey-Osborne 是这个赛道所有后来者的参照系——它第一次把"自动化暴露"做成了可引用的数字表。方法是让一组 ML 研究者在 70 个职业上手工打分（能/不能被自动化），然后用随机森林外推到剩下 632 个。

jobs 与它的对比是方法论的代际跳跃：打分从人类专家（70 人）换成 LLM（Gemini Flash，342 次调用）、打分维度从"自动化概率"换成"AI 暴露程度"、基础数据从 2010 年 O*NET 换成 2024 BLS OOH。

更深的对比在发布姿态。Frey-Osborne 的论文发了就发了，作者无法控制"47%"被做成多少个 BBC 头条；jobs 直接在 README 第一屏写"This is not a report"，并且公开标注"LLM rough estimates"。这是把 10 年媒体反噬的教训内化为产品设计。

用户视角：Frey-Osborne 的被引次数超过 12,000 次，但作者 Osborne 2024 年的后续论文里明确说"我们不再用那个 47% 的数字"——学术界已默认那个数字的意义被媒体消耗殆尽。jobs 的立场是"不给你一个数字，给你一个工具"，方向上走的是反向路线。

vs. OpenAI "GPTs are GPTs" (2023)

Eloundou et al. 2023 年的论文（由 OpenAI + 宾大）是 LLM 时代第一份正式 "exposure paper"。它的做法是把 O*NET 的 20k 个任务切开，让 GPT-4 和人类专家各自标注"该任务是否可被 GPT-powered tool 至少减半时间"，最后按职业聚合。

jobs 的方法是极简版：342 个职业（不是 20k 任务），一个 0-10 分（不是两层标注），一个 LLM（不是双盲专家对照）。 精度上 jobs 肯定输；但在"可分叉重跑"上它赢——任何人 fork，换一个 prompt.md 里的 rubric，几分钟跑一遍就能拿到自己视角的分数。

生态位：OpenAI 那篇是"科研参照点"，jobs 是"玩具探索器"。两者功能不重叠——如果你要写论文引用，选 OpenAI；如果你要在自己机器上体验"假如我换个标准会怎样"，选 jobs。

vs. Anthropic Economic Index

Anthropic 2024 年起每季度发布的 Economic Index 方法最激进：直接用 Claude 真实对话记录，聚类出"每个 O*NET 职业的 LLM 使用强度"，并追踪时间序列。

这是"事后观测"而不是"事前预测"——jobs 的 Gemini 打分问的是"AI 会不会重塑这个职业"，Economic Index 问的是"此刻有多少 Claude 调用真的发生在这个职业里"。 两者应该配合看：Economic Index 告诉你"现在"，jobs 给你"可能的 trajectory"。

用户视角：Economic Index 在 AI 研究圈和经济学圈有真实引用，但它有一个硬伤——只能看到用 Claude 的人。企业内部用 AI 的闭源场景（如 Google 内部、Microsoft Copilot 使用）它完全看不到。jobs 的 LLM 打分虽然粗，至少覆盖所有 342 个职业、不偏一家厂商。

vs. McKinsey / Goldman Sachs / WEF 报告

这三家每半年发一份 "AI impact on jobs" 的 PDF，数字一份比一份夺眼球（"300M jobs at risk"、"7.2M jobs created by 2027" 等）。它们的共同特征：非开源、非结构化数据、按行业 narrative 写，读者只能接受或否认。

jobs 正好是它们的反面：全部数据开源、结论可 toggle、方法论 60 行 Python 写完。 这不是"谁更权威"的问题，而是"信任模型不同"——McKinsey 让你信任品牌，jobs 让你信任代码。

vs. OECD PIAAC-based estimates

OECD 2016-2018 一系列基于 PIAAC 任务调查的报告，得到了"只有 14% 的 OECD 工作高自动化风险"——比 Frey-Osborne 低很多，因为他们按任务层聚合回职业，而不是整职业打分。

这条线最值得借鉴的东西jobs 没有做：按任务切分。jobs 的 LLM 拿到的是整职业 Markdown 描述，LLM 自己在内部做任务分解。如果 jobs 未来想严肃化，把 BLS OOH 里已有的 "Duties" 段落切出来、每个 duty 单独打分、再聚合回职业，就能得到一个 OECD-style 的细粒度版本。这是"工具→方法论"升格的最直接路径。

2.3 用户视角（真实反馈）

live demo karpathy.ai/jobs 上线后，HN 和 Twitter 上的可追溯讨论聚焦在三点：

  • "软件开发 9/10 但 BLS 预计 +25%" — 最被讨论的悖论。这条反馈直接催生了 exposure vs outlook 视图。
  • "能不能换 robotics exposure？" — README 里专门回应："you could write a different prompt for any question — e.g. exposure to humanoid robotics, offshoring risk, climate impact"。这句话是对工具化立场最直接的背书。
  • "没权重、没 confidence" — 一些读者指出 Gemini Flash 打分会有偏见。作者选择承认而不修补，放在 README 的 "NOT" 段落里。

2.5 机会与风险

  • 最大机会：成为 BLS 数据的"事实上的浏览器"。任何人想讲一个关于美国就业的 story，能 fork jobs 改 prompt / 改配色 / 截图。当前 BLS 官方网站在移动端/可视化上非常糟糕，这块空白实在大。
  • 最大风险：LLM 打分的 bias 被放大引用。一旦有人把"jobs 上医生是 5 分"当作"卡帕西说医生风险中等"拿去传，README 里的免责条款拦不住。
  • 次要风险：BLS OOH 每隔 2 年大更新，jobs 的抓取脚本/HTML 结构假设会失效。当前没有 CI，下次 BLS 改 UI 这个工具会静默坏掉。

三、横纵交汇总结

3.1 路径依赖判断

jobs 的 11 条 commits 给它锁定了三个结构性特征：

  • "工具而非结论"的定位被写进 README 第一屏。这条线未来几乎不可能撤回——一旦有用户把它当"报告"引用被作者纠正过公开一次，这个定位就成了项目的法律文本。
  • LLM-as-audience 的 prompt.md。一旦这个文件存在，后续所有维护都要同步更新它——否则你会有一个前端漂移版本 + 一个 LLM 可对话但过时的快照。这种"双产出"是长期维护负担，但也是差异化价值。
  • 单一 AI 暴露维度（0-10, digital-only）。这条定义的克制是优点也是天花板——想升级成 OECD-style 任务切分版，需要的不是小改而是重构 score.py。

3.2 差距的可追赶性

差距可追赶性推理
学术严谨度（vs OpenAI / OECD）部分可追赶按 duty 切分 + 双模型对比评分 + 加 variance 报告，可拉近差距，但要 3-5 倍工作量
真实使用信号（vs Anthropic Economic Index）不可追赶这需要自有 LLM 产品的对话日志；jobs 是第三方工具永远拿不到
媒体权威（vs McKinsey）不值得追赶追上去就违反自己的工具定位
BLS 数据完整度已满342 个职业就是 BLS OOH 全量
视觉设计 / 交互可追赶当前是 d3 式静态 treemap，空间还大

3.3 决策建议

  • 最应该做的一件：把 pipeline 和 prompt 抽成一个"BLS 探索模板"——让 "fork & rerun with custom rubric" 从 README 的一句话变成一个具体的 template repo（"karpathy/bls-explorer" 之类），再在此基础上加 2-3 个示范 rubric（humanoid robotics、offshoring risk、burnout risk），这样工具化定位会被广泛复制。当前 jobs 还停在 single-rubric 的示例阶段，稍显浪费。
  • 最不应该做的一件：不要把 AI 暴露分数推成榜单/周报/交互追踪。一旦开始定期发布 "本月 AI 暴露变化榜"，jobs 就滑向 McKinsey/Goldman 的叙事位；而它建立在可信度极低的 rough LLM estimate 上，这种 drift 会在 3-6 个月内把它的公信力吃光。永远不要为了更多流量而承担"你说的话会被当结论"的风险。
